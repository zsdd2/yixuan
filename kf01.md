## 2. 开发前必须完成

1. 明确用户最终目标，业务逻辑不确定时先确认。
2. 根据任务读取对应导航文档，不需要通读所有历史报告。
3. 检索相关前端页面、后端路由、Schema、ORM 模型和现有接口。
4. 若发现旧逻辑、旧路径或文档矛盾，先标注风险，再改代码。

## 3. 核心铁律

### 3.1 后端

- Router 必须使用 `APIRouter(prefix="/api/v1/xxx", tags=["xxx"])`。
- 数据库 Session 和当前用户必须通过 `Depends()` 注入。
- CPU/磁盘重任务必须用 `BackgroundTasks`、`run_in_threadpool` 或服务层调度器。
- 具体路由必须写在通配路由前，尤其是含 `/{token}` 的审核/交付模块。
- ORM Model 与 Pydantic Schema 必须分离。
- 修改模型或枚举必须新增 Alembic 迁移并验证迁移版本。

### 3.2 前端

- API 调用统一走 `frontend_admin/src/api/request.ts`。
- 登录接口固定为 `/api/v1/auth/login`。
- Vite 开发端口固定为 `15173`，保持 `strictPort: true`。
- 公开页面仅限 `/share/:token` 与 `/delivery/:token`。
- 管理员页面必须在路由 meta 中声明角色限制。

### 3.3 资产安全

- NAS 根路径来自 `NAS_MOUNT_PATH`，禁止硬编码绝对路径。
- `/storage/{file_path}` 必须限制在 NAS 根目录内。
- 客户审核页面只允许返回 `thumbnail_path`，严禁返回 `original_path`。
- 公开 token 页面新增字段前必须检查资产泄露、过期、禁用状态。

### 3.4 文档同步

| 改动 | 必须同步 |
| --- | --- |
| 新增/修改页面 | `docs/PAGE_BUSINESS_FLOW.md` |
| 新增/修改接口 | `docs/API_NAVIGATION.md`、`API_REGISTRY.md` |
| 新增通用规范 | `docs/STANDARDS.md`、`kf01.md` |
| 修改架构/状态机/模型 | `ARCHITECTURE.md` |
| 重大修复或审计 | `docs/AUDIT_*.md` |

## 4. 废弃路径

以下路径不得新增调用；若浏览器出现这些请求，优先检查旧前端进程、缓存或遗留页面：

- `/api/v1/system/auth/login`
- `/api/v1/system/users`
- `/api/v1/system/images`

## 6. 最重要的 10 条

1. 先定位业务流程，再改代码。
2. 不猜业务逻辑。
3. 不在错误逻辑上继续打补丁。
4. 不破坏 FastAPI 依赖注入。
5. 不阻塞 async 路由。
6. 不让通配路由盖住具体路由。
7. 不在客户页面暴露原图路径。
8. 不新增旧 `/api/v1/system/*` 伪路径。
9. 改接口就同步 API 文档。
10. 改模型就同步 Alembic 迁移。
