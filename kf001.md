# 通用开发准则文档

文档性质：跨项目开发入口规范  
适用范围：Web 应用、管理后台、业务系统、API 服务及相关前后端项目  
最后更新：2026-05-23  
当前版本：v1.1

本文件用于在任何项目开发、修复、重构、部署前统一工作方式。它不绑定具体业务、技术栈或目录名；进入某个项目后，应结合该项目自己的 `README`、`ARCHITECTURE`、`API`、`docs/`、迁移脚本和测试说明使用。

## 0. 首次使用适配流程

第一次把本准则用于某个项目时，先做一次“项目适配”，再开始开发。适配的目的不是改写本准则，而是建立当前项目的导航索引、验证命令、发布方式和风险清单。

适配流程：

1. 识别技术栈：前端框架、后端框架、数据库、ORM、迁移工具、构建工具、部署方式。
2. 建立目录映射：页面目录、组件目录、接口封装、后端路由、模型、Schema、服务层、迁移、部署配置。
3. 建立文档映射：入口文档、架构文档、API 文档、业务流程文档、部署文档、审计或变更记录。
4. 建立验证命令：前端构建、后端语法检查、测试、迁移、健康检查、容器配置校验。
5. 建立发布规则：版本号、镜像/发布包生成方式、数据备份、升级步骤、回滚步骤。
6. 建立项目红线：不能破坏的接口、状态流转、权限边界、数据目录、公开访问规则。
7. 形成“项目适配卡片”，放在本文末尾或项目自己的规范文档中。

适配原则：

- 项目适配卡片可以包含具体路径、端口、命令和账号说明。
- 通用准则正文不应因为某个项目而收窄适用范围。
- 项目专属规则必须标注来源，例如来自架构文档、部署文档、代码现状或用户确认。
- 当项目结构变化时，只更新适配卡片和项目文档，不随意改动本准则的核心原则。

项目适配卡片模板：

```text

```

## 1. 快速导航原则

接手任务时，不要从全量通读开始，应按任务类型精准定位资料。

| 当前任务 | 优先查看 | 补充查看 |
| --- | --- | --- |
| 页面、交互、用户流程 | 页面说明、产品文档、业务流程文档 | 相关组件、路由、状态管理 |
| 接口、认证、响应格式 | API 文档、接口注册表、请求封装 | 后端路由、Schema、错误处理 |
| 数据库、枚举、状态机 | 架构文档、模型定义、迁移目录 | 服务层、数据初始化脚本 |
| 部署、环境、容器 | 部署文档、Compose/K8s 配置、环境变量模板 | Dockerfile、启动脚本、日志策略 |
| Bug 修复 | 复现路径、日志、最近变更 | 测试用例、历史修复记录 |
| 重构 | 架构文档、模块边界、依赖关系 | 测试覆盖、兼容要求 |
| 安全、权限、公开页面 | 权限模型、认证流程、公开接口清单 | 日志、审计、访问限制 |

## 2. 开发前必须完成

1. 明确用户最终目标，而不是只理解表面修改点。
2. 确认影响范围：页面、接口、模型、权限、数据、部署是否会被影响。
3. 检索已有实现，优先沿用项目现有模式，不重复造轮子。
4. 业务逻辑不确定时先确认；技术细节不确定时先阅读代码和文档。
5. 发现文档、代码、数据库现状不一致时，先记录风险，再设计修改方案。
6. 修改前确认工作区是否已有他人改动，不覆盖无关变更。
7. 能小范围修复就不做大范围重构；确需重构时先保证测试和回滚路径。

## 3. 核心开发准则

### 3.1 通用原则

- 代码必须围绕真实业务流程组织，而不是围绕临时页面补丁堆叠。
- 每次修改都要有明确边界：改什么、不改什么、为什么。
- 不在错误抽象上继续叠加逻辑；必要时先拆清旧逻辑。
- 不把配置、密钥、环境路径、外部地址硬编码进业务代码。
- 不把临时脚本、测试数据、缓存文件、构建产物混入正式发布包。
- 不引入与项目风格冲突的新框架、新库或新目录结构，除非收益明确。
- 所有用户可见错误都应可理解；所有系统错误都应可排查。

### 3.2 后端准则

- 路由、服务层、数据模型、请求/响应 Schema 应保持职责分离。
- 数据库连接、当前用户、权限上下文等共享资源应通过统一依赖或中间件注入。
- 认证、授权、角色判断必须集中处理，不在多个业务函数里复制散落。
- 具体路由应优先于通配路由注册，避免动态参数吞掉固定路径。
- CPU、磁盘、网络、图片处理、批量导入等耗时任务不得阻塞主请求线程。
- 公开接口必须检查鉴权、过期、禁用、资源归属和数据泄露风险。
- 修改模型、枚举、索引、约束、默认值时，必须同步数据库迁移。
- 迁移脚本要兼容新环境和已有环境，避免重复执行失败。
- 数据初始化必须幂等：重复运行不能生成脏数据，也不能覆盖真实业务数据。
- 日志应记录关键上下文，但不得输出密码、Token、密钥、隐私数据。

### 3.3 前端准则

- API 请求应经过统一请求封装，集中处理 baseURL、Token、错误提示和登录失效。
- 页面状态、表单状态、筛选状态、弹窗状态应有清晰边界，避免互相污染。
- 管理页面、公开页面、客户页面、内部工具页面必须有明确路由和权限声明。
- 用户操作应有反馈：加载中、成功、失败、空状态、禁用状态。
- 批量操作必须显示选中数量、目标状态、风险提示和完成结果。
- 删除、覆盖、作废、权限变更等高风险操作应二次确认或可撤销。
- 图片、文件、长列表应考虑懒加载、分页、固定尺寸和失败占位。
- 组件优先复用现有设计系统和项目通用组件，不随意新增视觉风格。
- 前端不得依赖后端未声明字段；接口变更必须同步类型定义和文档。

### 3.4 数据与迁移准则

- 数据库结构以迁移脚本为准，不靠手工改库维持运行。
- 新增字段应明确 nullable、默认值、索引、唯一性、外键和删除策略。
- 状态字段应有清晰枚举和状态流转规则，不使用散乱字符串。
- 软删除、归档、物理删除要明确区别，并同步查询过滤逻辑。
- 重要数据修改应尽量可追踪：创建人、更新时间、操作来源、审计记录。
- 生产迁移前必须备份数据库；迁移失败要有回滚或修复方案。
- 初始化数据和演示数据必须分离，生产环境不得自动导入演示数据。

### 3.5 文件与资产安全

- 所有文件访问必须限制在允许的根目录内，防止路径穿越。
- 上传文件必须限制类型、大小、扩展名和存储位置。
- 文件名、路径、URL 参数不得直接信任用户输入。
- 私有文件与公开文件要分区存储或通过权限校验访问。
- 公开页面只返回必要资源，不暴露内部路径、原始文件、敏感元数据。
- 删除数据库记录时，应明确是否同步删除物理文件。

### 3.6 部署与配置准则

- 应用镜像应尽量无状态，业务数据放到数据库、卷、对象存储或外部目录。
- 生产环境必须使用环境变量或密钥管理配置敏感信息。
- 默认账号、默认密码、默认密钥不得作为生产长期配置。
- Dockerfile、Compose、启动脚本必须能在干净环境中复现部署。
- 发布包应排除 `.env`、日志、缓存、测试素材、依赖目录和构建临时文件。
- 每次发布应有版本号、变更说明、升级步骤和回滚方式。
- 数据库、上传文件、业务资产应分别制定备份策略。

## 4. 文档同步要求

| 改动类型 | 必须同步 |
| --- | --- |
| 新增/修改页面 | 页面说明、路由说明、业务流程文档 |
| 新增/修改接口 | API 文档、请求/响应示例、错误码 |
| 新增/修改模型 | 架构文档、迁移说明、字段含义 |
| 新增权限规则 | 权限矩阵、角色说明、公开接口清单 |
| 新增部署方式 | 部署文档、环境变量模板、回滚说明 |
| 重大 Bug 修复 | 修复说明、影响范围、验证方式 |
| 新增通用规范 | 本准则或项目规范文档 |

## 5. 废弃与兼容策略

- 废弃接口、字段、页面或配置时，应标记废弃原因和替代方案。
- 不得在新代码中继续调用已废弃路径。
- 对外部客户、第三方系统、旧版本客户端有影响的改动，应保留兼容期。
- 发现旧调用仍在发生时，优先定位来源：缓存、旧前端、旧任务、第三方调用。
- 删除兼容逻辑前，应确认日志中已无旧调用或业务方已完成迁移。

## 6. 验证闭环

每次修改至少完成对应层级验证。

后端常规验证：

```bash
python -m compileall app
alembic current
alembic upgrade head
curl http://localhost:8000/health
```

前端常规验证：

```bash
npm run build
```

容器部署验证：

```bash
docker compose config
docker compose up -d
docker compose ps
docker compose logs --tail=100
```

功能验证：

```text
1. 打开核心页面
2. 完成登录或鉴权流程
3. 执行本次修改涉及的主流程
4. 检查失败提示、空状态、刷新后状态
5. 检查浏览器控制台和后端日志
```

## 7. 发布前检查

1. 工作区无误提交的密钥、日志、缓存、测试数据。
2. 构建通过，迁移通过，核心流程通过。
3. 环境变量模板完整，生产值不写入仓库。
4. 发布包或镜像可在干净环境启动。
5. 数据库和文件资产已有备份。
6. 版本号、变更说明、升级步骤、回滚步骤齐全。
7. 默认账号和默认密钥已要求上线后修改。

## 8. 最重要的 12 条

1. 先理解业务目标，再设计技术改动。
2. 不猜业务规则，不猜数据状态。
3. 不覆盖无关改动，不混入无关重构。
4. 不硬编码密钥、路径、域名、端口和生产配置。
5. 不阻塞主请求线程处理重任务。
6. 不让通配路由覆盖具体路由。
7. 不在公开页面暴露敏感数据和内部路径。
8. 改接口就同步 API 文档和前端类型。
9. 改模型就同步迁移和架构说明。
10. 改权限就同步权限矩阵和验证用例。
11. 发布前必须验证构建、迁移、核心流程。
12. 生产更新前必须备份数据库和业务文件。

项目红线：

- 登录、权限、公开分享、图片访问、交付下载属于高风险流程，修改前必须先确认完整调用链。
