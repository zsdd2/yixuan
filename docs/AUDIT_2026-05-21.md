## 3. 主要发现

### P1：文档入口分散

根目录存在多份规范、审计、修复文档，开发者需要人工判断先读哪份，容易漏读 API 台账或架构约束。

处理：新增 `docs/README.md`，并把规范、页面流程、API 导航、审计记录分开。

### P1：遗留接口路径需要明确废弃

历史页面或缓存曾请求 `/api/v1/system/auth/login`，当前源码已使用 `/api/v1/auth/login`。`SettingsCenter.vue` 仍包含 `/api/v1/system/users`、`/api/v1/system/images` 等旧路径，但该文件当前未挂载路由。

建议：后续删除 `SettingsCenter.vue`，或把其中仍有价值的功能迁移到 `SettingsBasic/Images/Templates/Tags/Users`。

### P2：用户管理入口重复

当前存在：

- `/admin/users` -> `AdminUserPanel.vue` -> `/api/v1/users`
- `/settings/users` -> `SettingsUsers.vue` -> `/api/v1/settings/users`

两者会造成权限、字段、交互逐渐分叉。

建议：确定一个主入口。保留 `/admin/users` 作为管理员入口，`/settings/users` 可以重定向或复用同一组件。

### P2：项目详情承载过重

`ProjectDetail.vue` 同时承担项目头部、目标看板、目标详情、溯源看板、最终交付入口、审核分享入口。后续新增功能容易继续堆在一个页面。

建议：保持当前用户体验，但文档中按业务动作划分组件所有权，避免改动时误触其他分支。

### P2：公开 token 页面需要单独安全清单

审核页和交付页无需登录，任何字段新增都必须检查资产暴露、token 过期、禁用状态。

已处理：审核页 schema 已移除 `original_path`。
